# PDF Layout Fix Plan — Overlap between Clinical Information and Signature Block

## Top-Level Overview

The prenatal ultrasound PDF generated by `frontend/src/components/reports/pdfDocument.ts`
has a layout defect where the **Clinical Information** section (findings, comments, notes)
overlaps the **signature and date block** at the bottom of the page.

**Root causes identified:**

1. `kvGrid` double-advances the Y cursor when a section ends on a perfectly complete row
   (pairs count is an exact multiple of cols). This wastes ~7 mm after Ultrasound Findings
   and Pregnancy Data.
2. The row pitch inside `kvGrid` is 7 mm — tighter values are readable and save space.
3. Biometry renders at 2 columns (8 rows for 15 pairs). A 3-column layout reduces it to
   5 rows, saving 3 rows of pitch.
4. The signature block is hard-coded to `PAGE_H - 28` regardless of content height, so
   any overflow causes a visual overlap with no fallback.

**Approach:** Three targeted changes to `pdfDocument.ts`, no other files touched.
The combined savings (~61.5 mm) move Clinical Information to end at ~y=231 mm, leaving
~38 mm of clear space before the signature at y=269 mm on a typical single-page report.
Sub-Task 3 provides a runtime safety net for reports with unusually long clinical text.

## Sub-Task 1 — Fix `kvGrid` double-advance bug + reduce row pitch

**Status:** [x] completed

**Intent:**
Two bugs share the same fix location in `kvGrid`:

- **Double-advance bug**: when the last row of a grid is a *complete* row (pairs count is
  an exact multiple of `cols`), the loop's internal flush already advances `rowY` to
  `rowBottom + 7`, then `return rowBottom + 7` adds another 7 mm on top. The net trailing
  space is 14 mm instead of the intended 7 mm. This wastes ~7 mm after Ultrasound Findings
  (6 pairs ÷ 3 cols) and ~7 mm after Pregnancy Data (6 pairs ÷ 2 cols).
- **Row pitch**: the hard-coded `7` in `rowY = rowBottom + 7` (loop) can be reduced to
  `5.5` without readability loss (8 pt text has ~2.8 mm natural height; 5.5 mm pitch gives
  2.7 mm breathing room). Applied to ~21 data rows this saves ~31 mm.

**Expected Outcomes:**
- Ultrasound Findings and Pregnancy Data no longer add a double trailing gap.
- All section grids are 1.5 mm more compact per row.
- Total space recovered: ~31 mm (pitch) + ~14 mm (double-advance fix) = ~45 mm.

**Todo List:**
1. In `kvGrid` at line 146, change `rowY = rowBottom + 7` → `rowY = rowBottom + 5.5`
2. In `kvGrid` at line 147, change `rowBottom = rowY` — no change needed here, already correct
3. Fix the return at line 152: replace `return rowBottom + 7` with
   `return col === 0 ? rowY : rowBottom + 5.5`
   — when `col === 0` the loop already flushed and set `rowY` to the next row start, so
   return `rowY` directly; when `col > 0` the last row was partial and was never flushed,
   so add the 5.5 mm pitch from `rowBottom`.

**Relevant Context:**
- File: `frontend/src/components/reports/pdfDocument.ts`
- Function: `kvGrid`, lines 107–153
- The `col` variable is in scope at the return site (line 152) — no refactoring needed.

## Sub-Task 2 — Increase Biometry column count from 2 to 3

**Status:** [x] completed

**Intent:**
Biometry has 15 pairs. At 2 columns that is 8 rows; at 3 columns it is 5 rows — saving
3 rows. At the new 5.5 mm pitch that is 3 × 5.5 = **16.5 mm** recovered. All other
sections keep their existing column counts (doppler=3, ultrasound=3, anatomy=3, pregnancy=2).

At 3 columns each column is `COL_W / 3 = 60.7 mm`. The value area is `60.7 × 0.54 = 32.8 mm`
which comfortably fits values like `37 mm (1st %ile)` without wrapping.

**Expected Outcomes:**
- Biometry grid renders in 5 rows instead of 8.
- No label truncation or value wrapping (value width sufficient for all known biometry strings).

**Todo List:**

## Sub-Task 3 — Dynamic signature position with page-2 safety net

**Status:** [x] completed

**Intent:**
Replace the hard-coded `const SIG_Y = PAGE_H - 28` with a dynamic value:
`Math.max(y + 12, PAGE_H - 28)`. When content fits on one page (expected after
Sub-Tasks 1+2) the signature pins to the bottom exactly as before. When clinical text
is unusually long and `y` pushes past the threshold, a new page is added and the
signature renders at the top of page 2 — readable rather than overlapping.

The footer page counter must also update: `Page 1 of 1` on single-page reports,
`Page 1 of 2` / `Page 2 of 2` on two-page reports. The footer on page 1 must also
be drawn before the page break.

**Overflow threshold:** if `sigY + 20 > PAGE_H` (signature + 20 mm footer clearance
won't fit on current page), add a page and reset `sigY = 30`.

**Expected Outcomes:**
- Single-page reports: signature stays pinned at y=269, behaviour identical to today.
- Overflow reports: signature appears cleanly at top of page 2; page 1 gets its own footer;
  page counter reads correctly on both pages.
- No overlap under any content length.

**Todo List:**
1. Remove the line `const SIG_Y = PAGE_H - 28` (line 394).
2. Replace it with dynamic computation:
   ```
   let sigY = Math.max(y + 12, PAGE_H - 28);
   let totalPages = 1;
   if (sigY + 20 > PAGE_H) {
     // Draw page-1 footer before the page break
     // ... (same footer drawing code, with "Page 1 of 2")
     doc.addPage();
     sigY = 30;
     totalPages = 2;
   }
   ```
3. Update all references from `SIG_Y` → `sigY` (lines 395–411).
4. Update the footer `'Page 1 of 1'` text on line 422 to use
   `totalPages === 1 ? 'Page 1 of 1' : 'Page 2 of 2'`.

**Relevant Context:**
- File: `frontend/src/components/reports/pdfDocument.ts`
- Signature block: lines 392–411
- Footer block: lines 413–422
- `PAGE_H = 297` is a module-level constant (line 8)
- `FOOTER_Y = PAGE_H - 8 = 289` — this is a local `const` inside the footer block;
  on a two-page report it refers to the current (second) page's bottom, which is correct.
